## Minty: Create engaging video content

**TLDR:** 🚀 We’re launching [**Minty**](https://minty.so)—a tool that transforms any text into video content from stock images, making it easier for creators and businesses to reach diverse audiences. We’d love your feedback!

### **Our Ask:**

🙌 Check out [**Minty**](https://minty.so), try it out, and share your thoughts. Your feedback will help us shape the future of this tool. You can try it by inputting a blog article you want a video for.

### **❌ The Problem:**

In our journey with **Axolo**, we realized that repurposing content into various formats—blogs, presentations, videos—can be time-consuming and complex. With different audiences preferring different content formats, there’s a missed opportunity to fully engage with them.

### **The Solution:**

✨ **Minty** makes creating videos from text with just three simple steps:

1. **Form:** 📝 Enter your text, choose a music track, set the tone, pick a social media, and select a voice-over.
2. **Storyboard:** 🎨 Minty creates an outline of what will be said that you can edit like a Powerpoint presentation, allowing you to edit the voice-over of your video.
3. **Video Editing:** 🎬 We transform the storyboard into a fully edited video, complete with stock image, voice-over, and subtitles. You can customize it further with your own footage or choose from our stock library.

## Axolo: Merge code faster 🐙 now on GitLab (and GitHub)

On top of GitHub Axolo is now also available on GitLab. Now every team who is on Slack and GitLab or GitHub can collaborate seamlessly on code reviews.

**🤯 Problem:**

Why pay engineers 100 thousand dollars to have them sit and wait for their code to be reviewed? Code review workflow is, more often than not, unproductive.

\
We’ve identified three major problems:

1. Time to review: Takes too long to be reviewed and it’s painful.
2. Context switching: Switching between code review tools and Slack to communicate is a time waster.
3. Documentation: It’s hard to document everything that is being said. Merge requests are being talked about in DM and private calls. There is no documentation, and it’s hard to know why a decision was made.

**🎯Developers lose precious focus time**

We estimate that developers waste at least 2 hours per week coordinating with one another on pull requests.

For a team of 20 that represents the work of 1 developer full-time. This is at least 140k$ per year.

**🏚 In-house solutions and bots are going OBSOLETE.**

1. Noizy - Alerts everyone and then gets ignored
2. Hard to maintain
3. Miss critical info: deployments, GitHub Actions, and checks.

**\
🐙 Introducing Axolo: we bring back focus to your developers:**

Companies using Axolo have seen a 40% decrease in average cycle time. 86% of developers who use Axolo on a daily basis say that it has improved their code review workflow.\
\
**How does it work?**

It’s simple: Axolo creates an ephemeral Slack channel for each code review, invites only the right people, and adds all contextual information needed (CI/CD, ticket link, code conflicts, code review comments, etc..). When the code is merged the channel gets archived silently.

## Axolo 🐙  GitHub-Slack two-way synchronization

**Problem?** GitHub pull request workflow is unproductive

1. Time to review: Dangling pull requests are waste of time and a source of frustration for developers. They cause the code development process to slow down and prevent developers from focusing on a new task ahead.  Engineers need to write DM + @channels to get the attention their work needs.
2. Context switching: Between GitHub, Slack, video calls, and their product management software, engineers are piling up their mental load with so many tools and notifications.
3. Documentation: Pull requests are being talked about in DM and private calls. There is no documentation, and it’s hard to know why a decision was made.

\
**Solution?** Axolo centralizes pull requests in contextual Slack channels

1. 1 PR = 1 Slack channel. Only invites reviewers and creators.
2. Get personalized and friendly reminders about stale pull requests.
3. The conversation is saved on GitHub.
4. The Slack channel is auto-archive when the PR is merged and you can also auto-leave PR channels after approval.
