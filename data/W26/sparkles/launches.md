## Sparkles - Lovable for existing projects

Hey, I’m **Dan** - Founder of **Sparkles (W26)**

![uploaded image](https://www.ycombinator.com/media/?type=post&id=97488&key=user_uploads/2590397/2522ae4d-df6f-4327-bdf4-03a05bc9493b)

**Sparkles is like Lovable for existing projects**, so even your hypothetical Sharon from Operations can change the button styling on the front page without going back and forth with the dev team.

[Sparkles launch video](https://youtu.be/v6u7wlGYPIU)

**The problem**

If you’ve ever tried onboarding a non-technical person onto Claude Code / Cursor / Codex, you’ve definitely heard at least one of these:

* “I think I pushed the `.env` to GitHub… is that bad?”
* “Terminal is asking for my password but nothing is showing—my keyboard broke.”
* “It tells me to run `git push --force`… is that like a Star Wars thing ?”
* “Do I run `npm install` every time I open the laptop?”

This is what happens when a non technical person interacts with the CLI and dev server. It is a non user friendly environment that takes time to learn.

**The Solution**

![uploaded image](https://www.ycombinator.com/media/?type=post&id=97488&key=user_uploads/2590397/943d834c-9a37-4152-b3ec-3c015f87bb9f)

Sparkles is a web platform that fully abstracts from local development. Your GTM/Designer/Marketing team don’t need to know what is git, what is a cli, what is a dev server etc. They will see a lovable-like UI with the preview and the chatbox, so they can ship the updates as fast as they can type ✨

When someone makes a change on Sparkles they have an option to “upload” the changes, which essentially creates a Github PR for your engineer to review.

We are working on Enterprise level rulesets for PR creation, commit batching etc, so you don’t have to review thousands of PRs if you have a bigger team.
